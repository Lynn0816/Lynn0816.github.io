Extreme Programming 極限開發

發表於 2022-05-08 分類於 敏捷開發
文章字數： 1.3k 所需閱讀時間 ≈ 1 分鐘

前言

  • 極限開發(Extreme Programming, XP)是敏捷軟體開發的方法之一，最早起源於1996年，宗旨是強調團隊合作的統整與順暢性，並且認為軟體需求的不斷變化是很正常的，與其在專案初始時費心架構好嚴謹的一套應變方法，不如在專案的任何週期去適應變化，能高效產出高品質的軟體，這樣的想法起初是為了彈性適應不同企業的開發環境。
  • 以下將說明Extreme Programming的基本行為、價值、原則與實踐。

四種行為(Activities)

軟體開發的時候運作的四種行為︰

  • Coding(開發)

    • 找出最適合的解決方案
    • 交流programmer之間的想法

  • Testing(測試)

    • Unit Test確定功能是否如預期執行。自動化測試去盡可能測試每一段程式碼，當測試通過才代表這一段程式碼完成，接著才繼續下一段的開發。

  • Listening(聆聽)

    • 聆聽並了解客戶實際的目的與需求

  • Designing(設計)

    • 好的設計避免系統內的互相依賴關係，因此改變系統的一個部分不會影響系統的其他部分是很重要的。

五大價值(Values)

  • Communication(溝通)

    • 使用簡單設計、頻繁溝通與回饋來取代複雜的開發文件。

  • Simplicity(簡單)

    • 鼓勵使用簡單的解決方案開始，也就是著眼於當下的需求做開發與設計，因為不確定的未來耗費資源可能導致不必要的風險。

  • Feedback(回饋)

    • 系統的缺陷可以透過Unit Test來檢測、驗收測試(客戶)與定期集成測試來獲得回饋。

  • Respect(尊重)

    • 團隊中的任何人都應當被重視，一同以高品質追求最佳解決方案為目標。

  • Courage(勇氣)

    • Extreme Programming是重視當前的需求，針對並解決問題。因此Programmer要習慣重構程式碼。勇於刪除過時的程式碼，無論當時花費了多少努力。

三大原則(Principles)

  • Feedback(快速回饋)

    • 與客戶不斷的交流，掌握系統的開發方向。

  • Assuming simplicity(簡單假設)

    • 考慮眼前的需求來完成任務，不做額外考量程式的重用性。

  • Embracing change(包容變化)

    • 不確定的因素總是存在，以自然的心態因應客戶的需求變化。

實踐(Practice)

  • Fine-scale feedback(精細回饋)

    • Pair programming: 兩個programmer一同工作，一個人coding、一個人審查code，互相交流溝通並輪流互換角色。
    • Planning game
    • Test-driven development: 先寫測試再開發，有助於釐清程式的設計。
    • Whole team

  • Continuous process(持續處理)

    • Continuous integration: 每個programmer在提交程式碼之前，必須通過所有的單元測試、整合測試。
    • Refactoring or design improvement: 增加程式碼的可讀性或簡化結構。
    • Small releases

  • Shared understanding(共識)

    • Coding standards
    • Collective code ownership
    • Simple design
    • System metaphor

  • Programmer welfare

    • Sustainable pace
